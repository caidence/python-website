The Basics

Before anything else, I'd like to show my favorite quote about multiple inheritance. Dusty Phillips says in his book Python 3 Object-oriented Programming "As a rule of thumb, if you think you need multiple inheritance, you're probably wrong, but if you know you need it, you're probably right." If you are using multiple inheritance, unless you know that you absolutely need it, there is probably a better way to design your program. Many professional programmers even warn against it. Try not to use multiple inheritance unless you absolutely need to!

Just because it can be sticky, doesn't mean that I won't write a tutorial on it; so here we go. In theory, multiple inheritance is simple; a class that inherits from more than one other class. The subclass that inherits from more than one class will inherit attributes from all classes that it inherits from. This can be easier said than done, there are a lot of details to pay attention to with multiple inheritance and it's almost impossible to implement without having planned it from the beginning.

Directly calling init

The first method that we will discuss to multiple inheritance is a little more obvious than the second. It is more readable, but is still considered improper. Honestly, in my opinion, both ways work just fine so it really comes down to the programmer on which version to implement. This first tactic initializes itself by calling all of the parents' __init__ methods directly in its own __init__ method by explicitly using the name of each parent.

1. class Parent_1:
2.     def __init__(self, value_1=0):
3.         self.value_1 = value_1
4.
5.     def change_val_1(self, new_value):
6.         self.value_1 = new_value
7.
8.
9. class Parent_2:
10.     def __init__(self, value_2=0):
11.         self.value_2 = value_2
12.
13.     def change_val_2(self, new_value):
14.         self.value_2 = new_value
15.
16.
17. class Parent_3:
18.     def __init__(self, value_3=0):
19.         self.value_3 = value_3
20.
21.     def change_val_3(self, new_value):
22.         self.value_3 = new_value
23.
24.
25. class Child(Parent_1, Parent_2, Parent_3):
26.     def __init__(self, value_1, value_2, value_3):
27.         Parent_1.__init__(self, value_1)
28.         Parent_2.__init__(self, value_2)
29.         Parent_3.__init__(self, value_3)
30.
31.     def addition(self):
32.         return self.value_1 + self.value_2 + self.value_3

The Child class is will be the main focus of this example. Child inherits from Parend_1, Parent_2, and Parent_3. Because of this, instances created of the child class will be able to call methods from all three parent classes. Notice the Child classes init method. In this method, it explicitly calls each init method from each of its parents, passing the correct argument to the correct parent.

Parent_1's init accepts one argument: value_1. Likewise, Parent_2 and Parent_3 both accept one argument in their initialization methods. When we call each parents' init method explicitly from the child's init, we have to manually pass the correct value to each parent in order properly initialize instances. Also, when we explicitly call each parents' init like this, we have to pass the instance to each parent by using self as the first argument. Let's look at an interactive session with this file to demonstrate some of the concepts.

>>> test = Child(1, 2, 3)
>>> test.value_1
1
>>> test.value_2
2
>>> test.value_3
3
>>> test.change_val_2(22)
>>> test.value_2
22
>>> test.addition()
25

First, we create an instance of Child called test and initialize its values: value_1, value_2, and value_3 to equal 1, 2, and 3 respectively. On the next few lines, we can see that test's instance variables are equal to 1, 2, and 3. Since test inherits from Parent_1, Parent_2, and Parent_3; it has access to all of the methods in each parent class. We see this by calling the method change_val_2 on test. change_val_2 is a method within Parent_2, but inheritance allows us to call it on instances of the Child class.

The addition method in the Child class simply adds all of the instance variables together. I created this method with the intent to demonstrate that despite variables being initialized in parent classes; they still belong to the child instance.

This method of initializing instance variables of child classes can get messy quickly. This is especially true when you have existing code and you want to add a class for a child to inherit from. You would have to modify the child class, parent class and possibly some of the existing parent classes that the child already inherits from! The second method for initializing child classes by using multiple parent init methods is more robust than this solution.

Initialization with super()

Remember the super method that we learned about previously? This works excellent with inheritance, but it can be a little tricky when multiple inheritance comes in to play. Particularly when you want to initialize a child using multiple parent classes. How does super know which arguments to send to which parents init method? Well, the short answer is; it doesn't. The long answer is; we can use keyword arguments to easily create an "inheritance chain" so that all arguments will be passed to each parents init method. When we do this, init will take out the arguments that it needs and pass the kwargs to the next parent until all variables are initialized. This is a really confusing concept, so let's restructure the class from the last example to use this initialization method.

1. class Parent_1:
2.     def __init__(self, value_1=0, **kwargs):
3.         print("Parent 1")
4.         super().__init__(**kwargs)
5.         self.value_1 = value_1
6.
7.     def change_val_1(self, new_value):
8.         self.value_1 = new_value
9.
10.
11. class Parent_2:
12.     def __init__(self, value_2=0, **kwargs):
13.         print("Parent 2")
14.         super().__init__(**kwargs)
15.         self.value_2 = value_2
16.
17.     def change_val_2(self, new_value):
18.         self.value_2 = new_value
19.
20.
21. class Parent_3:
22.     def __init__(self, value_3=0, **kwargs):
23.         print("Parent 3")
24.         super().__init__(**kwargs)
25.         self.value_3 = value_3
26.
27.     def change_val_3(self, new_value):
28.         self.value_3 = new_value
29.
30.
31. class Child(Parent_1, Parent_2, Parent_3):
32.     def __init__(self, **kwargs):
33.         super().__init__(**kwargs)
34.
35.     def addition(self):
36.         return self.value_1 + self.value_2 + self.value_3

Take a look at each init method of Parent_1, Parent_2, and Parent_3. What I want you to look at is, each init method accepts one keyword argument, along with **kwargs. Also, each parent method passes the **kwargs argument to the super function. We'll get to why this works in a minute. Now look at the init method of the child class. See how this method does not explicitly accept any arguments other than kwargs. The Child class only accepts keyword arguments and then immediately passes kwargs to the super function.

This is where the order of parent classes comes in to play. In short, yes, the order of the classes that Child inherits from does matter. On line thirty-one, you can see that Child inherits from Parent_1, Parent_2, and Parent_3. This line is what I am referring to when I talk about the "order" of parent classes that Child inherits from. The order of the names in the inheritance chain does matter. In this case, Child will first inherit from Parent_1, Parent_2, then Parent_3. Because of this, technically Parent three does not need to call super().__init__(**kwargs) in its initialization function. However, because we still put this line in Parent_3's init method, the order of parents that Child inherits from is arbitrary.

To make this program work, we need to pass keyword arguments when creating instances of the Child class. Child's init function will immediately pass these arguments to be initialized in the three parent classes. Since Parent_1 comes first in the inheritance chain on line thirty-one, the arguments will all be passed to Parent_1's init method. Parent_1 will then strip any arguments that it needs (in this case, value_1) and then pass the rest of the arguments to the initialization function of the next listed parent class on line thirty-one. In the case of this example, that is Parent_2. Parent_2 will then do the same thing as Parent_1. It will take any values from **kwargs, initialize them accordingly, then pass the rest of the keyword arguments to the next parent's init method. This is why technically, the last parent class in the inheritance chain does not need the statement super().__init__(**kwargs). Because there is not another parent to pass keyword arguments to. If, however, we do include this statement in the last parent, it will essentially make the order of parents that Child inherits from arbitrary. This is because each parent class will now pass unneeded keyword arguments to the next parent.

Usually, the init method is not a place where you will see print statements. I decided to include the print statements in this example so that you can see the order of inheritance when we create an instance of Child. The order of inheritance is the same as the order of parent classes in the inheritance list on line thirty-one. This is probably as you'd expect, but I still feel it will help to show exactly how this works.

>>> test = Child(value_1=1, value_2=2, value_3=3)
Parent 1
Parent 2
Parent 3
>>> test.addition()
>>> test.change_val_1(10)
>>> test.addition()
15
>>> test.change_val_2(5)
>>> test.addition()
18

As we can see here, whenever we create an instance of Child, it will also call the print statements in each of its parents' init methods. In doing this, we can also see the order of which parent's init methods are called. In this example, the order is Parent_1, Parent_2, then Parent_3. This is because that is the order that Child inherits from the parents on line thirty-one. This program, functionally, works exactly the same as the last program that we looked at. The difference between the two is: this is the proper way to initialize child classes by using multiple parent classes and this method is more robust than the last.
