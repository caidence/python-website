The Basics

Both str and repr are meant to be string representations of objects, so what is the difference? Well, the goal of str is to be readable and the goal of repr is to be unambiguous. repr is used to compute the formal string representatio of an object where str is used more informally. Both of the built-in functions str() and print() really, behind the scenes call the __str__ method of an object. The repr() built-in method calls __repr__ behind the sense.

The str method is used to print out a "user-friendly" string that represents the object. This is the informal string representation of an object. The repr method is considered the formal string representation of an object. The string returned by the method should work as valid code to recreate the object that you call it on. This means that the string that is returned by repr should recreate the object if you call the eval method on it. In short, __str__ is more for users, __repr__ is more for developers.

str

The str method is more to tell what the object is and give some informal information about an object. For instance, you could print the location in memory, what class the object was created from, and even the arguments the instance variables of the object. It does not matter too much, str is just to give some general info about the object.

1. class Testing:
2.     def __init__(self, val_1, val_2):
3.         self.val_1 = val_1
4.         self.val_2 = val_2
5.
6.     def __str__(self):
7.         template = '<Testing object at {}; args: {}>'
8.         return template.format(id(self), self.__dict__)

The str method can really be any representation of the object that you want. Whenever you print or call the str built-in method on an object, if the class that the object was created from has a __str__ method, this method will be called. In the example, I decided to put the object's class name in the __str__ method, along with its location in memory, followed by its dictionary of instance variables. __str__ was meant to represent an object in an informal way. Many programmers will at least put the class name in the output of str, but there can be as much or as little information as you want. Here is a short interactive session with the class we created.

>>> t = Testing(1, 2)
>>> str(t)
"<Testing object at 2417952178976; args: {'val_1': 1, 'val_2': 2}>"
>>> print(t)
<Testing object at 2417952178976; args: {'val_1': 1, 'val_2': 2}>

repr

The repr method is more formal than the str method. The purpose for repr is to be the "actual" string representation of an object. This is more for developers than users. Both repr and str return a string representing an object. The string returned by repr should be work as valid code to create the same object that you called it on. A good way to test this is by using the eval built-in function. If you call eval(repr(object)), it should create an exact copy of the object.

1. class Testing:
2.     def __init__(self, val_1, val_2):
3.         self.val_1 = val_1
4.         self.val_2 = val_2
5.
6.     def __repr__(self):
7.         return 'Testing(val_1=%s, val_2=%s)' % (self.val_1, self.val_2)

In this example, we replaced __str__ from the last example with __repr__. You do not have to do this; any class you create can have both a str and repr method. I just am showing one at a time to hopefully alleviate some confusion. The repr method will return a string representing the current object. Notice how the string that repr will return looks exactly how we would create an instance of an object? This is what I mean that repr should return a string that would act as valid code. You want this method, if possible, to return a string that would create the object the same way that its current state is when you call the repr method. This method, in this example, will format val_1 and val_2 from the current instance and pass them as keyword arguments to the Testing class within the string. Remember, repr will return a string, not valid code. Here is what an interactive session with this class might look like.

>>> t = Testing(1, 2)
>>> repr(t)
'Testing(val_1=1, val_2=2)'
>>> t2 = eval(repr(t))
>>> repr(t)
'Testing(val_1=1, val_2=2)'
>>> repr(t2)
'Testing(val_1=1, val_2=2)'

We start by creating one instance of Testing and passing 1 and 2 to be the instance variables. When we write: repr(t), this will call the__repr__ method in our Testing class. Do you see how the output looks exactly the same as how we created the instance? This is by design, that is what repr is used for. If we pass the string returned by repr to the eval built-in function, it should run as valid code and essentially create a copy of the instance that we used __repr__ on. We can see how this works by setting a variable – t2 – equal to the evaluated string representation of t object. As we can see, now t2 is an exact copy of t.
